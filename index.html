1. Codebase Overview

1.1 Project Purpose

The Qubetics Chain Abstraction project implements a Multi-Party Computation (MPC) network for cross-chain cryptocurrency transactions. Key features include:

  • Distributed Key Generation (DKG) for shared key management
  • Threshold ECDSA signing across multiple nodes
  • Cross-chain bridging between Bitcoin and EVM chains
  • Liquidity pool management for solvers
  • VRF-based node selection for consensus

1.2 Architecture Components

Component Location Purpose
DKG Node `src/dkg/` Distributed key generation using Feldman VSS
Signing Node `src/signing/` Threshold ECDSA signature generation
Chain Handlers `src/chain_abstraction/` Bitcoin and Ethereum transaction handling
Sequencer `src/network/sequencer.rs` Transaction ordering and consensus
RPC Server `src/rpc_server/` External API for user operations
Network Layer `src/network/` P2P communication via libp2p

CRITICAL-02: Transaction ID Divergence (Consensus Failure) CVSS: 9.1
Location src/network/sequencer.rs:291-298
Status ‚úÖ VERIFIED - ACTIVELY USED
Impact Split-brain consensus, double execution

Vulnerable Code

// src/network/sequencer.rs:291-298
let final_transaction_id = if pending.contains_key(&transaction_id) {
    // Duplicate detected: append sequence to make it unique
    let unique_id = format!("{}:seq{}", transaction_id, sequence);
    warn!("Duplicate transaction_id detected: {}. Using unique ID: {}", transaction_id, unique_id);

Attack Scenario

Nodes process transactions in different orders. The local sequence counter is not synchronized.

Node A: Receives T1 first ‚Üí ID: T1:seq1
Node B: Receives T1 second ‚Üí ID: T1:seq2
Result: Nodes propose different IDs for same transaction ‚Üí Consensus Split

6. Verification Methodology

Each vulnerability was verified using the following process:

  1. Code Existence: Confirm the vulnerable code pattern exists using file reads and grep.
  2. Call Path Analysis: Trace from entry points to vulnerable code.
  3. Active Usage Confirmation: Determine if actively used, dead code, or partially used.
  4. Impact Assessment: Evaluate actual security impact (Exploitability, Scope, Availability).

7. Recommendations

Immediate Actions (CRITICAL) - P0

  • Hardcoded Nonce: Replace with RFC 6979 deterministic nonce generation.
  • Default API Key: Remove fallback, require explicit configuration.
  • Transaction ID Divergence: Use content-based hashing without local sequence.
  • Unanimous Consensus: Change to 2/3+1 threshold for completion.

Short-term Actions (HIGH) - P1

  • Data Deletion: Add production mode flag, remove auto-deletion.
  • CORS: Configure specific allowed origins.
  • Sleep Workaround: Implement proper event-driven synchronization.
  • API Key Logging: Remove credential logging entirely.
  • Placeholder Address: Fail explicitly when not configured.
  • Unwrap Usage: Replace with proper error handling.

8. Conclusion

This codebase is NOT suitable for production use. The hardcoded ECDSA nonce alone allows complete fund theft after 2 transactions.

Recommended Actions:

  1. Immediately halt any production/testnet deployment.
  2. Prioritize fixing CRITICAL-01 (nonce).
  3. Redesign consensus.
  4. Perform formal security audit.
